# 阶段 11：用户体验成熟化

## 1. 阶段目标

在不扩大 Agent 能力范围、不削弱安全边界的前提下，把已完成的 V1 功能整理为可理解、可发现、可高效操作的桌面 VS Code Extension。阶段以 [用户体验规范](../../ux.md) 为体验事实来源，以 [Webview Guidelines](../../webview.md) 为实现约束。

本阶段优先解决首次使用、当前对话、输入、Tool/审批、会话恢复和跨主题适配，不把“体验优化”作为协议扩张或后续能力开发的入口。

## 2. 前置条件与顺序

- 阶段 0–10 已完成，现有 Agent Loop、审批、命令、Checkpoint、错误分类和发布基线保持有效。
- 任务按 T1101 → T1108 顺序执行；一次只领取一个任务。
- 每个实现任务开始前检查是否改变协议、持久化、安全语义或公共组件契约。需要改变时先走变更控制，不在 UI PR 中顺带修改。
- 涉及 React、Zustand、Testing Library 或 VS Code API 时，执行任务前通过 Context7 获取当前文档。

## 3. 任务规格

### T1101：完成体验审计与场景基线

**目标**：对当前可安装版本执行启发式审计和核心路径走查，把体验规范中的要求映射到实际界面问题，并建立可复现的验收基线。

**产物**：

- 首次对话、普通聊天、只读 Tool、文件审批、命令审批、取消、错误和恢复路径的审计记录。
- 约 300px/常规宽度、四类 VS Code 主题、200% 缩放和纯键盘操作的基线结果。
- 带严重度、复现步骤、所属后续任务和截图或录屏证据的问题清单。
- 确认 `docs/ux.md` 与实际产品能力一致；发现规格偏差时先更新文档。

**测试**：在干净 VS Code Profile 与受控测试工作区执行人工走查；记录版本、主题、宽度和结果，不把主观描述作为唯一证据。

**排除项**：修改 Webview、协议、持久化或 Extension 行为；顺手修复审计发现的问题。

### T1102：建立视觉 Token 和基础组件

**目标**：统一 Webview 的间距、排版、边框、焦点、按钮、表单、Notice、Card 和 Empty State，减少组件之间的重复样式和视觉漂移。

**产物**：

- 基于 VS Code CSS Variables 的语义 Token。
- 当前阶段确有使用者的基础展示组件；不创建预测性组件库。
- 现有按钮、表单和状态提示迁移到统一交互状态。

**测试**：组件覆盖默认、hover、focus-visible、disabled、error 和 high-contrast 状态；验证 reduced motion。

**排除项**：更换样式技术栈、引入通用 UI 框架、品牌重设计、改变协议或业务流程。

### T1103：重构 Webview 信息架构

**目标**：让当前对话和输入成为默认视觉焦点，把会话历史、Checkpoint、设置和诊断组织为次级入口，并让上下文状态就近呈现。

**产物**：

- 新的页面 Shell、语义 landmarks 和窄侧栏布局。
- 无内容时隐藏或收起无关管理区域。
- 审批、错误和 Tool 状态与所属消息或当前 Run 建立清晰位置关系。

**测试**：组件测试覆盖空闲、运行中、待审批、失败和存在 Checkpoint 的布局；人工验证约 300px 宽度和 200% 缩放。

**排除项**：新增删除、归档、重命名会话等协议能力；改变 Extension 状态所有权。

### T1104：完善首次使用和配置引导

**目标**：用户无需先阅读 README，即可理解当前配置状态、完成必要设置并发起第一次成功对话。

**产物**：

- 区分 ready、缺少凭据、缺少模型、无效端点和不受信任工作区的欢迎/配置状态。
- 与当前能力一致的示例任务和明确设置入口。
- 配置完成后保留草稿并刷新可用状态。

**开始前约束门禁**：如果 Webview 现有安全 DTO 无法表达配置就绪状态，先提出最小协议变更及替代方案；不得把设置值、凭据或原始异常传入 Webview。

**测试**：覆盖每种配置状态、设置入口、配置后恢复和草稿保留；在隔离 Profile 中完成人工首次使用测试。

**排除项**：自动选择或探测 Provider、显示密钥内容、云端账户和遥测。

### T1105：改善对话阅读与 Composer

**目标**：提高长回复、代码内容和多轮对话的可读性，并让发送、换行、取消、重试和草稿行为可预测。

**产物**：

- 安全的消息富文本呈现与代码复制体验。
- `Enter` 发送、`Shift+Enter` 换行和可访问的操作提示。
- 输入框自适应、发送/停止/重试状态和草稿保留。
- 用户离开底部后的非强制自动滚动和返回最新内容入口。

**开始前约束门禁**：引入或扩展 Markdown 渲染能力前记录允许语法、链接处理、HTML 禁用/清理、代码长度边界和依赖选择；新增依赖必须有当前用例并更新 lockfile。

**测试**：覆盖键盘提交、输入法组合事件、草稿、流式滚动、取消、重试、长代码、恶意 HTML/链接和复制反馈；人工验证文本选择不会被流式更新重置。

**排除项**：消息编辑、队列、多模态附件和新的模型能力。

### T1106：简化 Tool 与审批反馈

**目标**：使用渐进披露呈现 Tool 生命周期，让用户先看到动作、影响和决策信息，再按需查看原始参数与输出。

**产物**：

- 面向用户的 Tool 摘要、紧凑成功状态和突出显示的运行中/失败/待审批状态。
- 文件修改与命令审批的信息层级，继续完整展示安全规范要求的不可变操作。
- 输出截断、取消、超时、拒绝、非零退出和错误的不同反馈。

**测试**：组件覆盖现有 Tool 和审批完整生命周期；协议夹具验证不得丢失安全关键信息；纯键盘完成展开、查看 Diff、批准、拒绝和终止。

**排除项**：改变 Tool 名称、风险等级、审批策略、命令执行语义或自动批准能力。

### T1107：改善会话、Checkpoint 与恢复体验

**目标**：让用户能够识别最近会话、理解中断状态，并把 Agent 修改与可用 Checkpoint 清晰关联起来。

**产物**：

- 基于现有 DTO 可提供信息的可扫描会话选择体验。
- 中断会话的恢复提示，以及活动 Run/草稿保护。
- Checkpoint 的目标、所属操作、时间、可恢复性和冲突反馈。

**开始前约束门禁**：若可识别标题、最近活动时间或其他需求需要持久化格式变化，先提出最小变更与“不改变格式”的替代方案，更新 `docs/persistence.md` 和 `docs/protocol.md` 后再实现。

**测试**：覆盖无会话、多个会话、活动 Run、中断恢复、Checkpoint 可恢复、Hash 冲突和用户后续修改；执行 Extension 人工恢复测试。

**排除项**：未单独批准的删除、归档、同步、自动标题或数据迁移。

### T1108：完成可访问性、主题与体验回归门禁

**目标**：对阶段 11 的完整体验执行跨主题、宽度、缩放、键盘、长内容和状态矩阵验收，并建立可持续的回归保护。

**产物**：

- 自动化可访问性检查和关键组件行为测试。
- 稳定场景的视觉回归基线；若现有工具链不足，先记录方案、确定性要求和维护责任再引入工具。
- 阶段验收矩阵、人工测试证据和剩余已知限制。
- 对 T1101 问题清单逐项给出已解决、明确接受或转入后续阶段的结论。

**测试**：运行 Webview 类型检查、组件测试和既有仓库检查；人工覆盖 `docs/ux.md` 第 9 节的全部场景。

**排除项**：借验收扩张产品能力、以截图差异替代交互断言、为了通过检查而降低可访问性规则。

## 4. 阶段门禁

- 新用户可以仅依据界面完成必要配置并发起首次成功对话，人工基线目标不超过 3 分钟。
- 默认空闲界面以当前对话和输入为主，不展示无内容的管理区域。
- 约 300px 宽度和 200% 缩放下，消息、输入和当前主要操作仍可使用。
- 普通聊天、只读 Tool、文件审批、命令审批、取消、错误和恢复路径均有验证证据。
- Light、Dark、High Contrast 和 High Contrast Light 主题通过验收。
- 主路径可以仅使用键盘完成，流式更新不抢焦点、不重置选择或强制滚动。
- UI 改动没有削弱安全、协议、取消、资源限制、持久化和模块边界。
- T1101 审计问题全部有明确结论，已知限制记录在发布检查材料中。
